Understanding how Orceum handles rate limits, timeouts, and retries helps you design apps that remain responsive under load and recover gracefully when things go wrong. This page covers the full lifecycle of a slow or rejected request.

## Timeout enforcement

Orceum waits **30 seconds** for a response from your app's endpoint. If your endpoint does not respond within this window:

- The action call is marked as failed
- The user receives an error message
- No retry is attempted — timeout failures are distinct from `429` retries

<Warning>
  The 30-second limit applies to the entire request-response cycle, not just processing time. Network latency between Orceum and your server counts toward the budget.
</Warning>

### Design for the 30-second limit

Most operations complete well within this limit. A few categories are unpredictably slow:

- File processing and exports
- Multi-step external API calls
- AI/ML inference pipelines
- Database report generation

For these, **return immediately** and process the work asynchronously:

```python
@app.post("/actions")
async def handle_action(request: Request):
    body = await request.json()
    event = body["event"]

    if event == "generate_report":
        task_id = await enqueue_report_job(body["event_data"])

        # Return immediately — don't wait for the job
        return {
            "result": {
                "task_id": task_id,
                "status": "processing",
                "message": "Your report is being generated. I'll notify you when it's ready."
            }
        }
```

Then push a webhook event when the job completes:

```python
async def on_report_complete(task_id: str, installation_id: str, report_url: str):
    await push_orceum_event(
        installation_id=installation_id,
        event_type="report.ready",
        event_data={
            "task_id": task_id,
            "report_url": report_url
        },
        priority="high"
    )
```

## Rate limiting (429)

When your app returns `429 Too Many Requests`, Orceum applies exponential backoff and retries the request up to three times before marking the call as failed.

| Attempt | Wait | Action |
|---------|------|--------|
| 1st | — | Original request |
| 2nd | Short backoff | Retry after delay |
| 3rd | Longer backoff | Retry after delay |
| 4th | — | Mark as failed, notify user |

Orceum makes up to **3 retry attempts** after the initial `429`. If all retries are rate-limited, the call fails and the user is informed.

### Retry-After header

If your app includes a `Retry-After` header, Orceum will honor it instead of using its own backoff schedule:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 60
Content-Type: application/json

{"detail": "Rate limit exceeded. Please wait 60 seconds."}
```

### Rate limit response headers

Return these headers alongside a `429` to give Orceum and your users precise context:

| Header | Description |
|--------|-------------|
| `X-RateLimit-Limit` | Total requests allowed in the current window |
| `X-RateLimit-Remaining` | Requests remaining in the current window |
| `X-RateLimit-Reset` | Unix timestamp when the window resets |
| `Retry-After` | Seconds to wait before retrying |

## Re-authentication (OAuth)

When OAuth tokens expire or become invalid, Orceum handles recovery automatically — no action is required from your app.

```
Action call → App returns 401
      │
      ▼
Orceum attempts token refresh
      │
      ├─→ Refresh succeeds → Retry call with new token
      │
      └─→ Refresh fails
              │
              ▼
        Installation → REAUTH_REQUIRED
        User notified with re-auth link
        User re-authenticates → Installation → ACTIVE
        Next action call proceeds normally
```

The re-auth link is generated by:

```bash
POST /v1/apps/{app_id}/installations/{installation_id}/reauth
```

This returns a URL the user clicks to restart the OAuth flow. Orceum handles the entire re-authentication process.

## API key re-submission

If `auth_type` is `API_KEY` and your app returns `401`:

1. Orceum marks the installation credentials as invalid
2. The user is prompted to re-submit their API key
3. The user provides a new key and the installation becomes `ACTIVE` again

Unlike OAuth, there is no auto-refresh for API keys — the user must actively re-submit. This is the expected flow when keys are rotated.

## Best practices

<AccordionGroup>
  <Accordion title="Return quickly, process async">
    Never perform heavy computation synchronously in your action handler. Enqueue the work, return a job reference immediately, then push a webhook when done. Users experience a fast, responsive app instead of watching a spinner.
  </Accordion>
  <Accordion title="Include clear rate limit headers">
    If you implement rate limiting, return `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers alongside `429`. Orceum surfaces these in error messages to give users context.
  </Accordion>
  <Accordion title="Design for idempotency">
    Because Orceum may retry requests on `429`, design your action handlers to be idempotent where possible. If the same `create_task` call is retried, it should not create duplicate tasks. Use the `X-Request-ID` header as a deduplication key.
  </Accordion>
  <Accordion title="Scale proactively">
    Monitor your `action_calls_last_30d` from the marketplace stats endpoint. If call volume is growing, ensure your rate limits and infrastructure can keep up — `429` errors degrade the user experience.
  </Accordion>
</AccordionGroup>
