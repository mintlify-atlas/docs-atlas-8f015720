Orceum uses two separate text fields to determine when to suggest your app and how to describe it. One is written for users; the other is written for the AI model. Getting both right directly affects how often your app is discovered and how accurately Orca AI uses it. This page explains what each field does, how to write them effectively, and how to complement them with action-level descriptions.

## Two audiences, two fields

| Field | Audience | Purpose |
|---|---|---|
| `description` | Users | Shown in the marketplace and installation screens — what users read |
| `app_summary` | Orca AI | Injected into the AI prompt — what the model reads before deciding to use your app |

## `description` — user-facing

The `description` is shown to users in the Orceum marketplace, installation screens, and conversation surfaces. Write it as you would an app store listing:

```json
{
  "description": "Manage your tasks and reminders. Create, update, and track todos with natural language — just tell Orca what you need done."
}
```

**Guidelines:**
- 1–3 sentences maximum
- Focus on what the user can *do* with your app
- Use plain language — no jargon or technical terms
- Mention the service by name (for example: "Sync with Todoist", "Manage Notion pages")
- Keep it under 200 characters — truncated descriptions perform worse in search rankings

## `app_summary` — AI-facing

The `app_summary` is injected into Orca AI's system prompt when it evaluates whether to use your app and when it constructs the execution context. This field is not shown to users.

```json
{
  "app_summary": "A task management app for creating, listing, updating, and deleting todo items. Best used when the user wants to track work, set reminders, or organize their schedule. Each task has a title, optional due date, priority level, and completion state. The installation_id in each request identifies which user's task list to operate on."
}
```

**Guidelines:**
- Describe what your app does in operational terms, not marketing terms
- List the key entities your app manages (tasks, events, documents, channels, etc.)
- Explain when the app *should* be used and when it should *not*
- Mention important conventions the AI needs to know (for example: "priority is 1–5", "dates are UTC ISO 8601")

<Tip>
  Think of `app_summary` as the briefing you give a new employee before they use your app. What do they need to know to operate it correctly from day one?
</Tip>

## Pre-install instructions

You can show users important information before they install your app. Use this for prerequisites, warnings, or setup steps they need to complete first:

```json
{
  "config": {
    "show_instruction_before_install": true,
    "instruction_before_install": "To use this app, you need a Todoist Pro account. Go to todoist.com/app/settings/integrations to generate an API token — you'll need it during setup."
  }
}
```

<ParamField body="show_instruction_before_install" type="boolean" default="false">
  When `true`, the installation UI displays `instruction_before_install` before the user clicks "Install".
</ParamField>

<ParamField body="instruction_before_install" type="string">
  Plain text (or light Markdown) shown to users before they begin installation.

  Use this for:
  - Prerequisites — "Requires Admin role"
  - Where to find credentials — "Go to Settings → API → Generate Key"
  - Important warnings — "This app will have read/write access to all your files"
  - Pricing notes — "Premium plan required"
</ParamField>

## Improving AI discoverability

Orca AI discovers apps based on the user's stated intent. The better your descriptions, the more accurately the model matches user requests to your app and its actions.

### Action-level descriptions

Each action's `description` in your manifest is equally important. Orca AI reads these to decide *which* action to call:

<Tabs>
  <Tab title="Too vague">
    ```json
    {
      "name": "create_task",
      "description": "Creates a task"
    }
    ```
    This tells the AI nothing useful about when to choose this action over any other.
  </Tab>
  <Tab title="Clear and actionable">
    ```json
    {
      "name": "create_task",
      "description": "Create a new task or to-do item with an optional due date and priority. Use when the user wants to remember something, add to their task list, or set a reminder."
    }
    ```
    Now the AI knows exactly when to call this action and what context it applies to.
  </Tab>
</Tabs>

### Parameter descriptions

Parameter descriptions are also read by the AI. Include format examples and valid ranges:

```json
{
  "name": "due_date",
  "type": "string",
  "description": "Due date in ISO 8601 format (YYYY-MM-DD), e.g. 2024-12-31. Leave empty for tasks with no deadline.",
  "required": false
}
```

### AI description checklist

- [ ] `description` reads like an app store listing — what can the user do?
- [ ] `app_summary` explains what entities your app manages and when to use it
- [ ] Each action's `description` says *when* to call it, not just *what* it does
- [ ] Parameter descriptions include format examples and valid ranges
- [ ] `additional_usage_info` in `config` captures any runtime behavior the AI needs to know
